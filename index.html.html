About Me

The Journey from
Analyst to Engineer

I started my career writing complex SQL queries to answer business questions — and I got very good at it. But I kept hitting the same wall: the data I needed was fragmented, dirty, or simply not flowing anywhere useful. So I started building the pipes myself.

Today I architect ETL pipelines with Apache Airflow, engineer historical data warehouses, and build AI-powered systems that make predictions — not just reports. My background in data analysis gives me something most engineers lack: I understand what the data means, not just how to move it.

I'm actively expanding into PySpark, Databricks, Snowflake, and Machine Learning, with a goal of becoming a Senior Data Engineer and Data Scientist who can own the full data stack — from raw ingestion to intelligent output.

🎯 Career Objective

To build production-grade data infrastructure and ML systems at scale — engineering pipelines that are reliable, observable, and intelligent. I want to work at the intersection of data engineering and applied AI, turning messy data into systems that create measurable business impact.

PROJECT · 002
Sales Incentive Calculation Engine
Monthly incentive calculations for a large sales force were done manually in Excel, taking 3+ days, prone to errors, and causing trust issues between finance and sales teams.
ARCHITECTURE
Sales Data → MySQL Procs → Rules Engine → Payout Report
Tech Stack
MySQLStored ProceduresTriggersExcelPython
Key Challenge
Incentive rules had 12+ slab tiers, product category multipliers, and exception cases — all of which changed monthly. Hard-coding was not an option.
Solution
Designed a configurable rules table driven by stored procedures. Finance updates slab parameters in a config table; procedures recalculate everything automatically on month-close trigger.
Result
↓ Calculation time from 3 days → 12 minutes · Zero manual errors for 8 consecutive months
⭐ View on GitHub → Production
PROJECT · 003
Inventory Historical Data Pipeline
Two MySQL source servers held inventory snapshots with no historical tracking. Business couldn't answer "what was stock levels last Tuesday?" — critical for demand planning.
ARCHITECTURE
MySQL ×2 → Airflow DAG → Pandas ETL → DWH
Tech Stack
AirflowMySQLPythonPandasSQL Server
Key Challenge
Data duplication from unreliable DELETE conditions on source timestamps, NaN/NaT/inf values breaking MySQL inserts, and column name escaping issues across two heterogeneous sources.
Solution
Anchored deletes to DATE(etl_loaded_at) = CURDATE(), built a clean() helper to sanitize NaN/NaT/inf values pre-insert, and escaped % characters in column names for Python string safety.
Result
↑ 100% data integrity · Full 18-month historical backfill · Zero duplicate rows since deployment
⭐ View on GitHub → Active DAG
PROJECT · 004
Data Warehouse Migration: Oracle → SQL Server
Legacy Oracle DWH was costly to maintain, had poor BI tool integration, and couldn't scale with growing reporting demands from 5 business units.
ARCHITECTURE
Oracle DWH → Schema Map → ETL Scripts → SQL Server
Tech Stack
OracleSQL ServerPythonSSISTableau
Key Challenge
Oracle-specific syntax (ROWNUM, NVL, CONNECT BY), implicit type differences, and 60+ dependent reports that couldn't go dark during cutover.
Solution
Systematic object-by-object migration with a parallel-run validation framework. Python scripts compared row counts and checksums between systems pre-cutover. Zero-downtime cutover using dual-write for 2 weeks.
Result
↓ 65% licensing cost reduction · ↑ 3× query performance · Zero data loss in migration
⭐ View on GitHub → Completed
PROJECT · 005 · FLAGSHIP
Self-Learning Jewelry Pricing Oracle
Jewelry pricing in the family business depended entirely on a jeweler's intuition — inconsistent, non-scalable, and impossible to audit. Gold rate volatility made static pricing sheets useless within hours.
ARCHITECTURE
Gold API → Oracle DB → ML Model → Pricing Engine
Tech Stack
PythonOracle DBScikit-LearnML RegressionREST APIPandas
Key Challenge
Feature engineering jewelry pricing from sparse historical sales data. The model had to account for gold weight, stone cost, making charges, margin tier, and live market rates — with a feedback loop to self-improve on actual sales.
Solution
Trained a gradient-boosted regression model on historical transaction data stored in Oracle. Live gold rate API feeds real-time market context. Model retrains weekly on new actuals, improving pricing accuracy continuously.
Result
↑ 23% margin improvement · Pricing consistency across 3 store branches · Model RMSE improved 18% over 3 months of self-learning
