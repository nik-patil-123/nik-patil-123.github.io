PROJECT · 002 · CHANDUKAKA SARAF
Inventory Historical Data Pipeline (Airflow)
Inventory snapshots from two MySQL servers had no historical tracking. The business couldn't answer time-based stock questions — critical for demand planning and branch audits.
ARCHITECTURE
MySQL ×2→ Airflow DAG→ Pandas ETL→ SQL Server DWH
Tech Stack
Apache AirflowMySQLPythonPandasSQL Server
Key Challenge
Data duplication from unreliable DELETE conditions on source timestamps, NaN/NaT/inf values breaking MySQL inserts, and column name escaping issues across two heterogeneous source servers.
Solution
Anchored deletes to DATE(etl_loaded_at) = CURDATE() for idempotent runs. Built a clean() helper to sanitize NaN/NaT/inf pre-insert. Fixed % character escaping in column names for Python string safety.
Result
↑ 100% data integrity · Full historical backfill achieved · Zero duplicate rows since deployment

SQL Optimization
How I Cut Query Time by 30%: A Real Production Story
The exact queries I rewrote, the indexes I added, and the execution plan analysis that revealed where the time was actually going. No theory — just what worked in production.

When I joined Chandukaka Saraf as a Data Analyst, one of the first things I noticed was that our Power BI dashboards weren't loading the way they should. Branch managers would refresh a sales report and just… wait. Some scheduled jobs that fed our reporting tables were also running long enough that data wasn't ready when the morning reports needed it. Nobody had measured exactly how bad it was — it was just accepted as "the dashboard is slow today." I decided to actually measure it instead of guessing.

Finding where the time was going

I pulled execution plans for the queries powering our most-used dashboards and the nightly jobs feeding them. A few patterns showed up fast: some queries pulled far more columns than the report ever used, a few were written as nested subqueries where a JOIN would let the optimizer do its job properly, and a couple of the worst offenders were missing indexes entirely on the columns they filtered or joined on.

What I changed

I prioritized the highest-impact queries first rather than rewriting everything at once, so I could measure each fix individually. Added targeted indexes (not blanket ones — that costs you on writes), rewrote the worst subqueries as CTEs or direct joins, and trimmed SELECT statements to only the columns actually used downstream.

The result

Average execution time improved by about 30% across the queries I touched. More importantly, morning jobs started finishing inside their window reliably, and the dashboards people checked first thing stopped being a daily complaint.

The lesson

Measure before you optimize, and fix the highest-impact queries first. Nothing exotic here — indexing, JOIN restructuring, and trimming bloat, applied with some patience and actual measurement instead of guessing.
